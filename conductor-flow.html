Conductor

End-to-end flow diagram and implementer setup guide — one plugin, either AI tool

One plugin, one flow. Conductor is a single installed plugin (plugin/ in this repo). GitHub Copilot and Claude Code both load the exact same skills, agents, hooks, and conductor-* CLI tools from it — the pipeline behaves identically either way. The only differences are which CLI you install and which CI platform runs it, covered in the setup tabs below.

3 Implementer setup

Same plugin, same skills. Pick the tool your team already uses — the pipeline behaves identically either way.

You need: A GitHub repo, Copilot CLI (npm install -g @github/copilot), an ADO project with a PAT, a Slack incoming webhook URL, and repository admin access.
1
Install the Conductor plugin
copilot plugin install --from https://github.com/deepu-roy/Conductor
This makes the skills, agents, hooks, and conductor-* CLI tools available — nothing is copied into your repo.
2
Run bootstrap
copilot --prompt "/bootstrap-project"
Scans the repo, generates shared/project/PROFILE.draft.md with cited evidence, and copies the CI templates into .github/workflows/.
3
Review and activate the profile
Fix every [NEEDS HUMAN INPUT] entry, confirm stack detection, then mv shared/project/PROFILE.draft.md shared/project/PROFILE.md.
4
Fill in guideline stubs
Edit shared/project/guidelines/*.md and shared/project/overrides/.
5
Create GitHub repository secrets
Settings → Secrets and variables → Actions:
COPILOT_TOKEN, ADO_PAT, SLACK_WEBHOOK_URL.
Also set repository variables: ADO_ORG_URL, ADO_PROJECT.
6
Enable the workflows
The five YAMLs in .github/workflows/ activate on push: ado-poller, design-gen, implement-story, pr-review, post-merge. Check the Actions tab to confirm all five appear.
7
No native ADO webhook needed
GitHub Actions can't receive ADO service hooks directly, so ado-poller polls ADO hourly for work items tagged ai:ready, dispatches design-gen for each, then flips the tag to ai:processing. Run it manually via workflow_dispatch to test without waiting.
8
Set branch protection rules
Settings → Rules → Rulesets: require 1 human reviewer on main and design/**, require status checks to pass, prevent direct pushes.
9
Commit and push
git add shared/project/ .github/workflows/ && git commit -m "Adopt Conductor" && git push
10
Tag your first work item
Write a clear description in an ADO Feature, tag it ai:ready, run ado-poller manually the first time, and watch the design PR appear.
Copilot CLI is newer than Claude Code. Expect some rough edges in headless mode — especially around agent delegation and auth in CI. Pin the CLI version in your workflows once you find one that works.
You need: An Anthropic API key, an ADO project with a PAT (Work Items RW, Code RW, Build RW), a Slack incoming webhook URL, and the claude CLI installed locally for bootstrap.
1
Install the Conductor plugin
claude plugin add --from https://github.com/deepu-roy/Conductor
This makes the skills, agents, hooks, and conductor-* CLI tools available — nothing is copied into your repo.
2
Run bootstrap
cd <your-repo> && claude -p "/bootstrap-project"
Scans the repo, generates shared/project/PROFILE.draft.md with cited evidence, and copies the CI templates into .azure-pipelines/.
3
Review and activate the profile
Fix every [NEEDS HUMAN INPUT] entry, confirm stack detection, then mv shared/project/PROFILE.draft.md shared/project/PROFILE.md.
4
Fill in guideline stubs
Edit files in shared/project/guidelines/ — coding standards, error handling, naming, API contracts. Add project-specific overrides in shared/project/overrides/.
5
Create ADO variable group
Project Settings → Library → New variable group conductor-secrets:
ANTHROPIC_API_KEY (secret), ADO_PAT (secret), SLACK_WEBHOOK_URL (secret).
6
Create four pipelines
Each pointing at its YAML in .azure-pipelines/: design-gen, implement-story, pr-review, post-merge.
7
Wire the native ADO Service Hook
Project Settings → Service Hooks → Work Item Updated → Filter: tags contain ai:ready → Action: trigger the design-gen pipeline, mapping workItemId. Azure Pipelines lives in the same ADO org, so no polling is needed here.
8
Set branch policies
On main and design/**: require 1 human reviewer, require PR build to pass, prevent direct pushes.
9
Commit and push
git add shared/project/ .azure-pipelines/ && git commit -m "Adopt Conductor" && git push
10
Tag your first work item
Write a clear description in an ADO Feature, tag it ai:ready, and watch the design PR appear within a few minutes.
First-run expectation: The design PR will likely need 1–2 revision comments from the tech lead. The slicing will probably be too fine. That's normal — correct in the PR, and the system learns from the overrides you add.

4 What actually differs between the two

Everything else is identical
Skills & agentsSame plugin/skills/ and plugin/agents/ — one open-standard definition, read by both runtimes.
Project layerBoth read shared/project/ — no duplication, no drift.
CLI tools & hooksSame conductor-* bin tools and the same two hooks (conductor-scope-guard, conductor-bash-guard).
Trigger mechanismGitHub Actions polls ADO hourly (ado-poller) since it can't receive ADO service hooks directly. Azure Pipelines is triggered natively by an ADO Service Hook — one pipeline fewer, no polling delay.
OrchestrationBoth fork contract-dev → backend-dev ‖ frontend-dev as separate CI jobs/stages for determinism, not as in-session subagent calls.
